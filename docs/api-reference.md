# ClassiRoute API Reference

Base URL: `http://localhost:8000` (development) or your deployed URL.

ClassiRoute exposes two authentication schemes: one for the dashboard (JWT) and one for API consumption (Virtual API Keys). The primary endpoint is an OpenAI-compatible chat completions API that routes requests to the best provider for each prompt.

## 1. Authentication

### 1.1 JWT Authentication (Dashboard)

Used by all dashboard routes: `/keys/*`, `/analytics/*`, `/users/*`, `/auth/me`.

Obtain a token via `/auth/login` or `/auth/register`. Include it in the `Authorization` header:

```
Authorization: Bearer <jwt_token>
```

**Token details:**
- Algorithm: HS256
- Expiration: 30 days (configurable via `JWT_EXPIRE_DAYS`)
- Payload contains `sub` (user ID) and `exp` (expiration timestamp)

### 1.2 Virtual API Key Authentication (Chat API)

Used exclusively by the `/v1/chat/completions` endpoint.

Each virtual key starts with the prefix `clr-`:

```
Authorization: Bearer clr-<64_hex_characters>
```

Keys are generated by the [Virtual Keys API](#3-virtual-keys-api) and stored as a SHA-256 hash. The raw key is shown only once at creation time.

## 2. Chat Completions API

The primary routing endpoint. Accepts an OpenAI-compatible request body, assigns the prompt to a tier (weak, mid, strong) based on complexity, and dispatches to the configured provider for that tier.

### POST /v1/chat/completions

**Authentication:** Virtual API Key (see [1.2](#12-virtual-api-key-authentication-chat-api))
**Rate limit:** 60 requests per 60 seconds per user

#### Request Body

```json
{
  "messages": [
    {"role": "user", "content": "What is the capital of France?"}
  ],
  "stream": false
}
```

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `messages` | array | Yes | -- | Array of message objects (OpenAI format) |
| `stream` | boolean | No | `false` | If `true`, returns a server-sent event stream |

Each message object:

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `role` | string | Yes | One of `"user"`, `"assistant"`, `"system"` |
| `content` | string | Yes | The message content |

#### Non-Streaming Response (stream: false)

```json
{
  "id": "chatcmpl-<uuid>",
  "object": "chat.completion",
  "created": 1710000000,
  "model": "gpt-4o-mini",
  "choices": [
    {
      "index": 0,
      "message": {
        "role": "assistant",
        "content": "The capital of France is Paris."
      },
      "finish_reason": "stop"
    }
  ],
  "usage": {
    "prompt_tokens": 12,
    "completion_tokens": 8,
    "total_tokens": 20
  },
  "x_tier": "weak",
  "x_confidence": 0.87,
  "x_model_used": "gpt-4o-mini"
}
```

| Field | Description |
|-------|-------------|
| `model` | The actual model that served the request (from the assigned tier) |
| `x_tier` | The tier assigned by the router: `"weak"`, `"mid"`, or `"strong"` |
| `x_confidence` | The routing confidence score (0 to 1) |
| `x_model_used` | The provider model that handled the request |

#### Streaming Response (stream: true)

Returns a Server-Sent Events (SSE) stream. Each event is a JSON object prefixed with `data: ` following the OpenAI streaming format.

```
data: {"id":"chatcmpl-<uuid>","object":"chat.completion.chunk","created":1710000000,"model":"gpt-4o-mini","choices":[{"index":0,"delta":{"role":"assistant"},"finish_reason":null}]}

data: {"id":"chatcmpl-<uuid>","object":"chat.completion.chunk","created":1710000000,"model":"gpt-4o-mini","choices":[{"index":0,"delta":{"content":"The"},"finish_reason":null}]}

data: {"id":"chatcmpl-<uuid>","object":"chat.completion.chunk","created":1710000000,"model":"gpt-4o-mini","choices":[{"index":0,"delta":{"content":" capital"},"finish_reason":null}]}

...

data: {"id":"chatcmpl-<uuid>","object":"chat.completion.chunk","created":1710000000,"model":"gpt-4o-mini","choices":[{"index":0,"delta":{"content":" Paris."},"finish_reason":null}]}

data: {"id":"chatcmpl-<uuid>","object":"chat.completion.chunk","created":1710000000,"model":"gpt-4o-mini","choices":[{"index":0,"delta":{},"finish_reason":"stop"}]}

data: [DONE]
```

The final message in the stream includes `x_tier`, `x_confidence`, and usage metadata.

#### Routing Behavior

1. **Feature extraction:** The router extracts 40+ features from the prompt (code indicators, mathematical notation, reasoning markers, language patterns, etc.).
2. **ML classification:** A pre-trained classifier predicts the optimal tier (weak/mid/strong) and confidence score.
3. **Confidence upgrade:** If confidence is below 0.60 and the tier is not already strong, the request is upgraded to the next higher tier.
4. **Heuristic fallback:** When no ML model is loaded, a rule-based heuristic assigns the tier based on a complexity score (<2.0 = weak, <4.0 = mid, >=4.0 = strong).
5. **Cascading fallback:** At runtime, if a provider call fails, the system retries with the next lower tier automatically.

#### Examples

**curl:**
```bash
curl -X POST http://localhost:8000/v1/chat/completions \
  -H "Authorization: Bearer clr-abc123..." \
  -H "Content-Type: application/json" \
  -d '{
    "messages": [{"role": "user", "content": "Write a Python function to sort a list"}],
    "stream": false
  }'
```

**Python (httpx):**
```python
import httpx

client = httpx.Client(base_url="http://localhost:8000")
response = client.post(
    "/v1/chat/completions",
    headers={"Authorization": "Bearer clr-abc123..."},
    json={
        "messages": [{"role": "user", "content": "Hello, who are you?"}],
        "stream": False,
    },
)
print(response.json()["choices"][0]["message"]["content"])
```

**Python (OpenAI SDK):**
```python
from openai import OpenAI

client = OpenAI(
    base_url="http://localhost:8000/v1",
    api_key="clr-abc123...",
)

response = client.chat.completions.create(
    messages=[{"role": "user", "content": "What is the meaning of life?"}],
    model="any",  # model field is ignored; routing decides
)
print(response.choices[0].message.content)
```

## 3. Virtual Keys API

All keys endpoints require JWT authentication and are rate-limited at 200 requests per 60 seconds per user.

### POST /keys/create

Create a new virtual key with three provider tiers.

**Authentication:** JWT

**Request body:**

```json
{
  "name": "My API Key",
  "weak_model": "gpt-4o-mini",
  "weak_api_key": "sk-...",
  "weak_base_url": "https://api.openai.com",
  "weak_provider_type": "openai",
  "mid_model": "gpt-4o",
  "mid_api_key": "sk-...",
  "mid_base_url": "https://api.openai.com",
  "mid_provider_type": "openai",
  "strong_model": "claude-3-opus-20240229",
  "strong_api_key": "sk-ant-...",
  "strong_base_url": null,
  "strong_provider_type": "anthropic"
}
```

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `name` | string | Yes | -- | A human-readable label for the key |
| `weak_model` | string | Yes | -- | Model for low-complexity prompts |
| `weak_api_key` | string | Yes | -- | API key for the weak tier provider |
| `weak_base_url` | string | No | `null` | Base URL for OpenAI-compatible providers |
| `weak_provider_type` | string | No | `"openai"` | Provider type: `"openai"`, `"anthropic"`, `"gemini"` |
| `mid_model` | string | Yes | -- | Model for medium-complexity prompts |
| `mid_api_key` | string | Yes | -- | API key for the mid tier provider |
| `mid_base_url` | string | No | `null` | Base URL for OpenAI-compatible providers |
| `mid_provider_type` | string | No | `"openai"` | Provider type |
| `strong_model` | string | Yes | -- | Model for high-complexity prompts |
| `strong_api_key` | string | Yes | -- | API key for the strong tier provider |
| `strong_base_url` | string | No | `null` | Base URL for OpenAI-compatible providers |
| `strong_provider_type` | string | No | `"openai"` | Provider type |

**Validation rules:**
- When `provider_type` is `"openai"`, `base_url` is required.
- All three providers are verified before the key is created. An HTTP request is sent to each provider's `/models` endpoint to confirm the model exists and the API key is valid.

**Response `201 Created`:**

```json
{
  "key": "clr-a1b2c3d4e5f6...",
  "key_id": "uuid-of-the-key",
  "name": "My API Key"
}
```

The `key` field is shown only once. Store it securely. Provider API keys are encrypted at rest using Fernet (symmetric encryption).

**Error responses:**
- `400 Bad Request`: One or more providers unreachable. Includes an `errors` array with per-tier details.

```json
{
  "detail": {
    "message": "One or more providers are unreachable. Key not created.",
    "errors": [
      "Mid tier: Model 'gpt-4o' not found. Available: gpt-4o-mini, gpt-4-turbo...",
      "Strong tier: Invalid API key (401 Unauthorized)"
    ]
  }
}
```

## 5. Analytics API

All analytics endpoints require JWT authentication.

### GET /analytics/summary

Aggregate statistics for the authenticated user.

**Query parameters:**

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `key_id` | string | `null` | Filter by specific virtual key |
| `days` | integer | `30` | Lookback window (1 to 365) |

**Response `200 OK`:**

```json
{
  "total_requests": 1523,
  "by_tier": {
    "weak": 890,
    "mid": 420,
    "strong": 213
  },
  "total_cost_usd": 12.45,
  "cost_saved_vs_always_strong": 38.20,
  "avg_latency_ms": 842.0,
  "success_rate": 0.97
}
```

| Field | Description |
|-------|-------------|
| `total_requests` | Total number of routed requests in the period |
| `by_tier` | Breakdown of requests per tier (0=weak, 1=mid, 2=strong) |
| `total_cost_usd` | Estimated total cost across all tiers |
| `cost_saved_vs_always_strong` | Estimated savings vs. sending everything to the strong tier |
| `avg_latency_ms` | Average response latency in milliseconds |
| `success_rate` | Fraction of requests that completed successfully (0 to 1) |

## 10. Error Codes

All error responses follow the FastAPI convention:

```json
{
  "detail": "Human-readable error message"
}
```

Some endpoints return structured detail objects with additional context.

| Status Code | Meaning | Common Causes |
|-------------|---------|---------------|
| `400` | Bad Request | Invalid input, email already registered, provider validation failure |
| `401` | Unauthorized | Missing or invalid authentication, expired JWT, inactive virtual key |
| `403` | Forbidden | Admin-only resource accessed by non-admin user |
| `404` | Not Found | User, key, or resource does not exist |
| `429` | Too Many Requests | Rate limit exceeded |
| `500` | Internal Server Error | Unhandled server exception |
| `502` | Bad Gateway | Provider API returned an error or is unreachable |

### 401 Error Details

**JWT authentication:**
```json
{
  "detail": "Invalid token format. Expected 'Bearer <token>'"
}
```

**Virtual key authentication:**
```json
{
  "detail": "Invalid API key format. Expected 'Bearer clr-...'"
}
```

**Invalid or inactive key:**
```json
{
  "detail": "Invalid or inactive virtual key"
}
```

### 429 Error Details

**Chat rate limit:**
```json
{
  "detail": "Rate limit exceeded. Retry after 45s"
}
```

Headers: `Retry-After: 45`

**Auth rate limit:**
```json
{
  "detail": "Too many auth attempts. Please wait before trying again."
}
```

Headers: `Retry-After: 12`

## 11. Rate Limiting

ClassiRoute uses a sliding window rate limiter. Each bucket tracks timestamps within the current window.

### Rate Limit Table

| Scope | Limit | Window | Key |
|-------|-------|--------|-----|
| Chat | 60 requests | 60 seconds | `chat:{user_id}` |
| Auth | 20 requests | 60 seconds | `auth:{ip_address}` |
| API (keys, users) | 200 requests | 60 seconds | `api:{user_id}` |
| Admin | 100 requests | 60 seconds | `admin:{user_id}` |

### Response Headers

Rate-limited endpoints include these headers:

| Header | Description |
|--------|-------------|
| `Retry-After` | Seconds to wait before retrying (integer) |

### Handling Rate Limits

When you receive a `429` response, wait for the number of seconds specified in the `Retry-After` header before retrying. Use exponential backoff for production applications.

## 12. Provider Configuration per Tier

Each virtual key defines three tiers. Each tier has its own model, API key, base URL, and provider type.

### Supported Provider Types

| Provider Type | Description | Base URL Required | Key Verification |
|---------------|-------------|-------------------|------------------|
| `openai` | OpenAI-compatible API (OpenAI, NVIDIA NIM, etc.) | Yes | Models endpoint |
| `anthropic` | Anthropic Claude API | No | Minimal message |
| `gemini` | Google Gemini API | No | Generate content |

### Tier Behavior

| Tier | Index | Typical Use Case | Example Models |
|------|-------|------------------|----------------|
| Weak | 0 | Simple Q&A, basic translation, factual lookups | `gpt-4o-mini`, `claude-3-haiku`, `gemini-2.0-flash-lite` |
| Mid | 1 | Code generation, reasoning, analysis | `gpt-4o`, `claude-3-sonnet`, `gemini-2.0-flash` |
| Strong | 2 | Complex reasoning, math, advanced coding | `gpt-4-turbo`, `claude-3-opus`, `gemini-1.5-pro` |

### Cascading Fallback

When a provider call fails for the assigned tier, ClassiRoute automatically falls back through lower tiers:

- Strong fails -> retries Mid, then Weak
- Mid fails -> retries Weak
- Weak fails -> returns error to client

The fallback is transparent: the response includes the original tier information in the metadata so you can see which tier was originally selected versus which tier ultimately served the request.

### Encryption

Provider API keys are encrypted at rest using Fernet (symmetric encryption with `ENCRYPTION_KEY`). They are decrypted only in memory at request time.
